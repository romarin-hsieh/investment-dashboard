### **防衝突機制詳解**

#### **1. 提交標籤防護**
```bash
git commit -m "Update precomputed technical indicators $(date -u +%Y-%m-%d) [skip ci]"
```
- **作用**: 防止預計算提交觸發新的部署工作流程
- **機制**: GitHub Actions 識別 `[skip ci]` 標籤並跳過 deploy.yml
- **重要性**: 避免無限循環 (預計算 → 提交 → 部署 → 預計算)

#### **2. 並發控制**
```yaml
concurrency:
  group: "pages"
  cancel-in-progress: false
```
- **作用**: 確保同時只有一個 Pages 部署在進行
- **策略**: 新部署會等待當前部署完成，而不是取消
- **好處**: 避免部署衝突和資源競爭

#### **3. 條件檢查**
```bash
if ! git diff --staged --quiet; then
  git commit -m "..."
  git push
else
  echo "No changes to commit"
fi
```
- **作用**: 只有在數據真正變化時才提交
- **好處**: 減少不必要的提交和網路流量
- **邏輯**: 檢查暫存區是否有變化

#### **4. 錯誤處理與重試**
```javascript
async computeIndicatorsWithRetry(symbol, attempt = 1) {
  try {
    // 計算邏輯
  } catch (error) {
    if (attempt < this.retryAttempts) {
      await this.sleep(this.retryDelay);
      return this.computeIndicatorsWithRetry(symbol, attempt + 1);
    }
  }
}
```
- **重試次數**: 最多 3 次
- **重試延遲**: 5 秒
- **批次控制**: 最多 3 個並發請求

### **數據一致性保證**

#### **1. 原子性操作**
- 所有同一日期的技術指標數據在同一次提交中更新
- 使用索引文件 (`latest_index.json`) 確保數據完整性
- 失敗的計算不會影響成功的數據

#### **2. 版本控制**
- 每日數據使用日期前綴: `2025-12-24_ASTS.json`
- 保留歷史數據便於回滾和比較
- 自動清理 7 天前的舊數據

#### **3. 快取協調**
```javascript
// 前端快取策略與預計算時間協調
const CACHE_DURATIONS = {
  TECHNICAL_INDICATORS: 60 * 60 * 1000, // 1 小時
  PRECOMPUTE_TIME: '22:00 UTC',         // 美股收盤後 1 小時
}
```
- 快取時間短於預計算間隔，確保數據新鮮度
- 預計算在交易日結束後執行，確保數據完整性

## 🎯 **觸發情境詳細說明**

### **A. 主要部署 (deploy.yml)**

#### **觸發條件**
1. **自動觸發**
   ```yaml
   on:
     push:
       branches: [ main ]
   ```
   - **情境**: 當代碼推送到 `main` 分支時
   - **頻率**: 每次 push 都會觸發
   - **典型場景**: 
     - 開發者合併 PR 到 main
     - 直接推送修復到 main
     - 其他工作流程的自動提交

2. **手動觸發**
   ```yaml
   workflow_dispatch:
   ```
   - **情境**: 管理員手動觸發部署
   - **使用場景**:
     - 緊急修復部署
     - 測試部署流程
     - 重新部署失敗的版本

#### **執行流程**
1. **代碼檢出** → 獲取最新代碼
2. **環境設置** → Node.js 18 + npm 快取
3. **依賴安裝** → `npm ci` (生產環境安裝)
4. **專案建置** → `npm run build` (生產模式)
5. **Pages 設置** → 配置 GitHub Pages
6. **檔案上傳** → 上傳 `./dist` 目錄
7. **部署執行** → 部署到 GitHub Pages

#### **權限需求**
```yaml
permissions:
  contents: read      # 讀取代碼
  pages: write        # 寫入 Pages
  id-token: write     # 身份驗證
```

### **B. 技術指標預計算 (precompute-indicators.yml)**

#### **觸發條件**
1. **定時觸發**
   ```yaml
   schedule:
     - cron: '0 22 * * 1-5'  # 週一到週五 UTC 22:00
   ```
   - **時間**: 美東時間 17:00 (UTC 22:00)
   - **頻率**: 每個交易日 (週一至週五)
   - **策略**: 美股收盤後 1 小時執行
   - **目的**: 確保當日交易數據完整後再計算

2. **手動觸發**
   ```yaml
   workflow_dispatch:
   ```
   - **使用場景**:
     - 測試預計算功能
     - 補充遺漏的計算
     - 緊急數據更新

#### **執行流程**
1. **代碼檢出** → 獲取最新代碼
2. **環境設置** → Node.js 18 + npm 快取
3. **依賴安裝** → `npm ci`
4. **目錄創建** → `public/data/technical-indicators/`
5. **數據計算** → 執行 `scripts/precompute-indicators.js`
6. **結果提交** → 自動提交計算結果到 repository

#### **自動提交機制**
```bash
git config --local user.email "action@github.com"
git config --local user.name "GitHub Action"
git add public/data/technical-indicators/
if ! git diff --staged --quiet; then
  git commit -m "Update precomputed technical indicators $(date -u +%Y-%m-%d) [skip ci]"
  git push
else
  echo "No changes to commit"
fi
```

**重要特性**:
- **條件提交**: 只有當數據有變化時才提交
- **跳過 CI**: 使用 `[skip ci]` 標籤避免觸發新的部署
- **時間戳記**: 提交訊息包含執行日期

## ⚠️ **已知問題與解決方案**

### **問題 1: 部署衝突導致 404 錯誤**

#### **問題描述**
- **現象**: 網站出現 404 錯誤，無法正常訪問
- **原因**: 技術指標預計算的自動提交觸發了新的部署，導致部署衝突

#### **根本原因分析**
1. **觸發鏈**: 預計算 → 自動提交 → 觸發部署 → 部署衝突
2. **時機問題**: 預計算在美股收盤後執行，可能與其他提交衝突
3. **並發問題**: 多個部署同時進行導致資源競爭

#### **解決方案**
1. **添加 [skip ci] 標籤**
   ```bash
   git commit -m "Update precomputed technical indicators $(date -u +%Y-%m-%d) [skip ci]"
   ```
   - 防止自動提交觸發新的部署

2. **並發控制**
   ```yaml
   concurrency:
     group: "pages"
     cancel-in-progress: false
   ```
   - 確保同時只有一個部署在進行

3. **條件檢查**
   ```bash
   if ! git diff --staged --quiet; then
     # 只有在有變化時才提交
   fi
   ```
   - 避免不必要的提交

### **問題 2: 時區配置**

#### **問題描述**
- **預期**: 美股收盤後 30 分鐘執行 (16:30 EST)
- **實際**: 設定為收盤後 1 小時 (17:00 EST)

#### **時區對照表**
| 時區 | 美股收盤時間 | 預計算執行時間 | UTC 時間 |
|------|-------------|---------------|----------|
| EST (冬令時) | 16:00 | 17:00 | 22:00 |
| EDT (夏令時) | 16:00 | 17:00 | 21:00 |

#### **建議調整**
```yaml
schedule:
  # 冬令時: UTC 21:30 = EST 16:30
  # 夏令時: UTC 20:30 = EDT 16:30
  - cron: '30 21 * * 1-5'  # 調整為收盤後 30 分鐘
```

## 🚨 **故障排除指南**

### **部署失敗**
1. **檢查建置日誌**: 查看 npm run build 是否成功
2. **依賴問題**: 確認 package-lock.json 是否最新
3. **權限問題**: 檢查 GitHub Pages 設置
4. **並發衝突**: 等待其他部署完成後重試

### **預計算失敗**
1. **API 限制**: 檢查是否超過 API 呼叫限制
2. **數據格式**: 驗證輸入數據格式是否正確
3. **儲存空間**: 確認 repository 儲存空間充足
4. **網路問題**: 檢查外部 API 連線狀態

### **自動提交問題**
1. **權限檢查**: 確認 GITHUB_TOKEN 權限
2. **分支保護**: 檢查是否有分支保護規則阻擋
3. **提交格式**: 確認提交訊息格式正確
4. **檔案變化**: 驗證是否真的有檔案需要提交
